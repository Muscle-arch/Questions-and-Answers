# 四川大学智能问答助手网站需求分析文档

## 1 引言

### 1.1 编写目的
本文档旨在详细描述四川大学智能问答助手网站的需求分析，为项目开发、测试和部署提供指导。通过明确项目目标、功能需求、技术架构和实施计划，确保项目团队能够准确理解需求并高效完成开发工作，最终交付一个满足校园信息服务需求的智能问答系统。

### 1.2 背景
随着信息技术的快速发展，高校信息化建设已成为提升校园服务质量的重要手段。四川大学作为国内知名高校，拥有庞大的师生群体和复杂的校园环境，传统的信息获取方式已无法满足师生的需求。因此，开发一个智能问答助手网站，为师生提供便捷的校园信息查询和导航服务，具有重要的现实意义。

### 1.3 定义
- **智能问答**：基于人工智能技术，能够理解并回答用户提出的问题
- **RAG**：检索增强生成（Retrieval-Augmented Generation），结合知识库检索和语言模型生成的技术
- **LLM**：大语言模型（Large Language Model），如DeepSeek等
- **JWT**：JSON Web Token，用于用户认证的安全令牌
- **SSE**：Server-Sent Events，服务器向客户端推送事件的技术
- **API**：应用程序接口（Application Programming Interface）

### 1.4 参考资料
- 《四川大学信息化建设规划（2024-2028）》
- 《FastAPI官方文档》
- 《Vue 3官方文档》
- 《高德地图API开发文档》
- 《DeepSeek API文档》

### 1.5 标准、条约和约定
- 遵循RESTful API设计规范
- 采用HTTPS协议确保数据传输安全
- 遵循前端开发最佳实践，确保代码质量
- 遵循后端开发规范，确保系统稳定性
- 所有时间字段采用ISO 8601格式
- 数据库设计遵循关系型数据库设计规范

## 2 项目概述

### 2.1 项目目标
构建一个基于人工智能技术的校园智能问答助手网站，实现校园信息的智能查询、导航路线规划等功能，提升校园服务质量和用户体验。具体目标包括：
- 提供智能问答功能，回答关于校园信息的各类问题
- 提供校园导航功能，规划校园内各地点的路线
- 实现用户管理和会话管理功能
- 构建和维护校园知识库
- 确保系统的安全性、可靠性和可扩展性

### 2.2 项目详细需求内容

#### 2.2.1 前端功能需求
| 功能模块 | 功能点 | 描述 |
|---------|--------|------|
| 登录注册 | 用户登录 | 提供用户名和密码登录功能，支持任意用户名密码登录（开发阶段） |
| | 用户注册 | 允许新用户注册账号 |
| 智能问答 | 问题输入 | 提供文本输入框，支持用户输入问题 |
| | 流式回答 | 实时显示AI回答，支持markdown格式 |
| | 会话管理 | 显示历史会话列表，支持切换和删除会话 |
| | 建议问题 | 提供常见问题建议，点击即可发送 |
| 校园导航 | 导航查询 | 支持自然语言输入导航需求，如"从望江楼到化工学院怎么走" |
| | 地图显示 | 在页面中显示导航路线和地图 |
| | 路线规划 | 提供详细的步行路线步骤 |
| 用户中心 | 个人信息 | 显示用户基本信息 |
| | 登录状态管理 | 管理用户登录状态，支持退出登录 |

#### 2.2.2 后端功能需求
| 功能模块 | 功能点 | 描述 |
|---------|--------|------|
| 用户认证 | 注册接口 | 处理用户注册请求，创建新用户 |
| | 登录接口 | 验证用户凭证，返回JWT Token |
| | 获取用户信息 | 根据Token获取当前用户信息 |
| 会话管理 | 获取会话列表 | 返回用户的历史会话列表 |
| | 创建会话 | 创建新的对话会话 |
| | 获取会话消息 | 返回指定会话的消息记录 |
| | 删除会话 | 删除指定会话及其消息 |
| 智能问答 | 流式问答 | 处理用户问题，返回流式回答 |
| | RAG检索 | 从知识库中检索相关信息 |
| | LLM调用 | 调用大语言模型生成回答 |
| 校园导航 | 路线规划 | 解析导航请求，返回路线信息 |
| | 地标识别 | 识别校园内的地标位置 |
| | 地图API调用 | 调用高德地图API获取路线数据 |

#### 2.2.3 非功能需求
| 需求类型 | 具体要求 |
|---------|---------|
| 性能要求 | 页面加载时间 < 2秒，流式回答响应时间 < 1秒 |
| 可靠性 | 系统可用性 ≥ 99.5%，故障恢复时间 < 30分钟 |
| 安全性 | 采用HTTPS协议，JWT Token认证，密码加密存储 |
| 可扩展性 | 支持知识库的动态更新，支持新功能模块的添加 |
| 兼容性 | 支持主流浏览器（Chrome、Firefox、Safari、Edge） |
| 易用性 | 界面简洁直观，操作流程清晰，响应及时 |

### 2.3 关键技术
- **前端技术**：Vue 3 + Vite、Pinia、Vue Router、Element Plus、Axios、高德地图 JS API、marked + highlight.js
- **后端技术**：FastAPI、JWT、SQLite/PostgreSQL、DeepSeek API、高德地图 API
- **部署技术**：Docker + Nginx
- **AI技术**：RAG检索增强生成、大语言模型调用

### 2.4 项目资源与环境要求
- **开发环境**：本地开发机器（Windows/macOS/Linux）
- **测试环境**：云服务器（1核2G内存，50GB存储）
- **生产环境**：云服务器（2核4G内存，100GB存储）
- **网络要求**：稳定的网络连接，支持外部API调用
- **软件要求**：Node.js 18+、Python 3.9+、Docker

#### 2.5.4 应当提供的服务

## 3 项目团队组织

### 3.1 组织结构
项目采用敏捷开发模式，团队组织结构如下：
- 项目负责人：1名，负责项目整体规划和协调
- 前端开发组：1-2名，负责前端应用开发
- 后端开发组：1-2名，负责后端服务开发
- 测试组：1名，负责系统测试
- 运维组：1名，负责系统部署和维护

### 3.2 人员分工
| 角色 | 职责 | 人数 |
|------|------|------|
| 项目负责人 | 项目规划、进度管理、资源协调 | 1 |
| 前端开发 | 前端应用开发、UI设计、用户体验优化 | 1-2 |
| 后端开发 | 后端服务开发、API设计、数据库设计 | 1-2 |
| 测试工程师 | 功能测试、性能测试、安全测试 | 1 |
| 运维工程师 | 系统部署、服务器管理、技术支持 | 1 |

### 3.3 协作与沟通

#### 3.3.1 项目团队内部协作
- 采用每日站会制度，同步项目进展和问题

- 定期召开项目评审会议，评估项目进展

#### 3.3.2 项目接口人员
- 项目负责人：作为项目对外接口，负责与相关部门沟通

#### 3.3.3 项目团队外部沟通与协作模式
- 与学校信息化部门保持定期沟通，获取需求反馈
- 与高德地图API、DeepSeek API等第三方服务提供商保持沟通

## 4 项目实施计划

### 4.1 风险评估及对策
| 风险类型 | 风险描述 | 影响程度 | 应对措施 |
|---------|---------|---------|---------|
| 技术风险 | LLM API调用失败 | 高 | 实现降级策略，当LLM不可用时返回预设答案 |
| | 地图API调用失败 | 中 | 缓存常用路线数据，减少API调用 |
| 数据风险 | 知识库数据不足 | 中 | 定期更新知识库，增加数据采集渠道 |
| | 数据安全问题 | 高 | 加密存储敏感数据，实现数据访问控制 |
| 部署风险 | 服务器资源不足 | 中 | 监控系统资源使用，及时扩容 |
| | 网络不稳定 | 中 | 实现请求重试机制，提高系统稳定性 |
